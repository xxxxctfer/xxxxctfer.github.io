                  学习一下nacos框架的东西

                  2026-04-09

                  naco是一个内网外网都很常见的框架，我们这里来学习一下平常的时候怎么打这种系统，避免打项目的时候还要现学。后面还有ruoyi框架的，敬请期待。 整了半天虚拟机跑服务，老是报错，直接开本地了，github上面下载源代码，然后直接解压再启动就行。 https://github.com/alibaba/nacos/releases/tag/1.4.0 然后Windows启动的话需要打命令：./startup.cmd -m standalone。然后我们就可以开始复现了。

                  基本情况

                  先简单了解一下这个框架相关的东西。

                  1

                  Nacos是一个更易于构建云原生应用的动态服务发现、配置管理和服务管理平台，由阿里巴巴开源。它致力于帮助您快速实现动态服务发现、服务配置、服务元数据及流量管理。Nacos支持几乎所有主流类型的服务发现、配置和管理，包括Kubernetes Service、gRPC & Dubbo RPC Service、Spring Cloud RESTful Service等。通过Nacos，您可以轻松构建、交付和管理微服务平台，实现服务的动态发现、配置和治理。

                  权限控制

                  是一个很正常的权限控制的地方。没什么特别的，用户管理，角色管理，权限管理三个东西。 但是可能存在泄露的可能性。但是这个是后面的东西了，这个不用太管现在，反正就是一个管理界面。

                  命名空间

                  简单看看，里面只有public一个命名空间。 这个有什么用呢，其实就是一个类似分类的地方，我们前面创建服务的时候，或者创建配置的时候，都会有一个分组相关的选项，其实就是代表了他们的命名空间，主要就是用来分类不同的东西，比如有客户端，服务端，销售组，采购组，财务组等等。都需要自己开创一个命名空间，好分辨。

                  集群管理

                  这个在单主机的情况下其实用处不大，简单来说这个就是几个实例在一起，被设定为一个集群。有什么作用呢，我们先来看看集群的显示。 可以看到有一个节点就是192.168.65.1，但是实际上这个是我本地的一个虚拟IP，和我本地IP，127.0.0.1，如果看作两个主机的话，他们之间就会形成一个集群，可以缓解服务的压力，比如A主机有注册服务，然后有很多请求同时处理，可能就会处理不过来。 这个时候集群就发挥作用，将A主机的服务，同步到B，C主机上来，然后由nacos来自动分配请求，由三个主机来处理这些请求。我们可以回去看看我们的服务，已经被同步了。 可以看到我们的服务，192.168.65.1也创建了一个临时实例，这个就是集群的作用。

                  基本情况总结

                  其实通过对功能的解析我们就可以知道，这个框架实际上就是一个可以配置微服务，由nacos自动处理请求，发送到对应的服务，由服务中的代码处理，自动去运行一些事件帮忙处理。现在再去看前面的介绍就可以明白了。

                  漏洞复现

                  然后这个框架很明显有很多历史漏洞，我们先来学习一下，有时间可以去进一步挖掘内部存在的漏洞。

                  Nacos默认配置未授权访问漏洞

                  这个是一个比较经典的漏洞，形成原因是因为默认未开启配置nacos.core.auth.enabled，我们来找一下这个配置在哪里。 路径为nacos/conf/application.properties，可以看到这个配置是false的。 影响版本为 Nacos 1.2.0 至 2.2.0 版本。

                  攻击测试

                  我们来测试一下poc。

2

                  http://192.168.65.1:8848/nacos/v1/auth/users?pageNo=1&pageSize=9
http://192.168.52.1:8848/nacos/v1/auth/users?pageNo=1&pageSize=9&search=accurate&accessToken

                  这两个都可以试一下，效果是差不多的。 但是这个利用方式的话，可能效果比较有限，因为这里的密码存储还是密文的，而且是哈希，单向加密，只能尝试碰撞可能出现，所以主要还是弱口令多一点。 但是这里不是只暴露了这个api，而是完全暴露，这里只是一个泄露而已，但是内部也是可以未授权访问的，相当于完全裸奔，需不需要密码区别已经不大了，可以未授权创建用户等等操作了。 直接curl都能创建，很明显了吧，我们直接去登录验证一下。

                  但是这里的环境比较粗糙，没有管理员的权限，但是也可以去创建一个专门的管理员，添加之类的操作。这个还是挺危险的。

                  漏洞原理

                  虽然刚刚讲了漏洞原理是因为配置文件里面的东西，但是还是找找代码里面是哪里设置了这个参数为false，就可以全局随便访问了。我们下载了一份源代码，直接来看一下。 nacos-1.4.0/core/src/main/java/com/alibaba/nacos/core/auth/AuthFilter.java 我们可以看到这个文件里面的doFilter()方法。 可以看到这里有一个authConfigs.isAuthEnabled()方法，我们去看看是啥。 nacos-1.4.0/auth/src/main/java/com/alibaba/nacos/auth/common/AuthConfigs.java 可以看到这里就直接去提取了配置文件里面的这个配置，简单分析下代码就知道是false的时候，就可以无限制访问了，原理还是比较简单的。

                  防御建议

                  很简单，把配置文件里面的nacos.core.auth.enabled改成true就行。

                  Nacos 认证绕过漏洞（CVE-2021-29441）

                  依旧是配置文件的问题，貌似跟留的后门似的。简单来说就是，配置文件中的nacos.core.auth.enable.userAgentAuthWhite = true。这里很难绷，1.4.1之后的版本才出现这个配置，在1.4.0版本里面是找不到这个东西的，但是是默认开启的，所以没事，我们试一下。 影响版本：

                  • Nacos 1.x < 1.4.1（1.0.0 ~ 1.4.0）
                  • Nacos 2.x < 2.0.0-RELEASE（早期 2.x 测试版）

                  攻击测试

                  我们这里需要先把前面那个漏洞的配置关闭，再重新启动一下，更新他的配置文件，不然就不知道是谁的漏洞导致的绕过了。

                  我们先测试一下前面的漏洞，确认没有未授权访问了。 OK，没问题了，没有未授权，我们现在加上UA头再测试一下。 OK，可以看到加上UA头之后，直接就进行了访问。那么很明显，前面的那些创建用户，分配管理员等等，都是可以去打的，这里就懒得打了，知道这个地方可能有这个漏洞就是。在1.4.0版本之前是一定有，除非修改了源代码，在1.4.0版本之后的话，就是配置问题，导致的漏洞出现。

                  漏洞原理

                  其实就在上面那个漏洞代码的下一行就是漏洞代码。 这里没有到别的地方找了，因为这个版本没有开启这个配置，而是写死的，所以我们可以通过UA头直接打这个漏洞，而不需要看配置文件。

                  防御建议

                  更新nacos版本，不要使用1.4.0版本以前的版本，使用后续版本也需要更改配置。

                  Nacos默认key导致权限绕过登陆（QVD-2023-6271）

                  这个描述也是有点长了，但是很明显，就是一个默认key的问题，配置文件问题真是多。我们来简单看看吧。 可以看到这里有一个专门的key，是什么作用等下再说，可以看到这里是写死的，也就是说默认key是公开的，我们就可以通过默认key去进行一个绕过。 影响版本：0.1.0 <= Nacos <= 2.2.0

                  攻击测试

                  我们其实随便看看就可以知道这里的鉴权是用jwt的，我们可以直接去伪造一个出来，https://www.jwt.io/，我们去生成一下时间戳。https://www.beijing-time.org/shijianchuo/ 然后把时间戳，用户名，还有key这些东西丢进去。这里很搞啊，原文章里面的功能现在找不到了，搞了有一段时间，都有点问题，干脆直接用脚本加密了。(因为这里的密钥需要进行一次base64解码，等会看代码的时候解释)

                  有点迷糊，这里有点难整，等后续我们找工具试一下吧，这里感觉很奇怪。 OK，可以看到这里直接显示出来了存在key的默认配置。 可以看到通过这个漏洞，我们可以在工具里面的权限绕过漏洞利用当中添，删，改用户。简单来说就是绕过来权限检测，可以以管理员进行操作，虽然我们不是管理员。

                  漏洞原理

                  我们找到跟JWT签名相关的类，即nacos-1.4.0/console/src/main/java/com/alibaba/nacos/console/security/nacos/JwtTokenManager.java 就可以看到这里有老熟人，authConfigs类，调用了getSecretKeyBytes，我们去看一下这个方法。

                  很明显这个secretKey是从配置里面拿的，所以这里必须要BASE64.decode一次，再放到这个bytes[]中去，所以我们需要先base64解码，再作为密钥传进去。

                  防御建议

                  修改Key不是默认密钥，尽可能复杂度高一些。

                  Nacos身份认证serverIdentity绕过

                  这个同样跟配置文件有关，感觉这个框架就把配置文件相关的东西，全部找一遍，写死的配置加上有问题的代码就是个漏洞了。 影响版本：

                  • 1.4.1 ≤ Nacos ≤ 1.4.4
                  • 2.0.0 ≤ Nacos ≤ 2.2.0 好巧不巧，这里正好是1.4.0的版本的，没有这个漏洞，但是我们后续也需要打2.0.0版本的漏洞，所以这里暂留一下，等到后续打的时候再来。 来看一下配置文件。 这里还可以看到之前的userAgentAuthWhite，就是之前那个1.4.0版本没有的配置，这里是有的，但是我们这里主要是看下面这两个key和value。这个也像一个后门，可以进行身份认证(其实都是为了集群模式的方便，但是确实很像给攻击者留的后门一样的) 我们这里为了打漏洞，把配置文件稍微改一下，然后key和value修改一下。

                  攻击测试

                  先看一下错误情况是什么样的。 回显是403，我们现在加上一个请求头，就是我们刚刚设置的，key就是头，value就是值。再看一下。 很明显，回显变成500，但是此时服务器是认为我们是合法访问者的，相当于绕过了鉴权，所以请求头还是有作用的。

                  漏洞原理

                  还是老朋友，但是换了个版本，还是把路径贴出来：nacos-2.0.0-bugfix/core/src/main/java/com/alibaba/nacos/core/auth/AuthFilter.java 就在前面UA头的漏洞点的下面，直接写上了这段逻辑，后续就不需要去看了。

                  防御建议

                  建议修改配置文件中的key和value，避免配置文件泄露，导致出现请求头被发现，被爆破出可能的"后门"。

                  @GetMapping(value = "/derby")  
public RestResult<Object> derbyOps(@RequestParam(value = "sql") String sql) {  
    String selectSign = "select";  
    String limitSign = "ROWS FETCH NEXT";  
    String limit = " OFFSET 0 ROWS FETCH NEXT 1000 ROWS ONLY";  
    try {  
        if (PropertyUtil.isEmbeddedStorage()) {  
            LocalDataSourceServiceImpl dataSourceService = (LocalDataSourceServiceImpl) DynamicDataSource  
                    .getInstance().getDataSource();  
            if (StringUtils.startsWithIgnoreCase(sql, selectSign)) {  
                if (!StringUtils.containsIgnoreCase(sql, limitSign)) {  
                    sql += limit;  
                }  
                JdbcTemplate template = dataSourceService.getJdbcTemplate();  
                List<Map<String, Object>> result = template.queryForList(sql);  
                return RestResultUtils.success(result);  
            }  
            return RestResultUtils.failed("Only query statements are allowed to be executed");  
        }  
        return RestResultUtils.failed("The current storage mode is not Derby");  
    } catch (Exception e) {  
        return RestResultUtils.failed(e.getMessage());  
    }  
}

                  可以看见就检测了一个要是select开头的一个查询语句，就可以直接拼接进行查询，那就是很明显的漏洞点了，不多介绍了。

                  防御建议

                  升级nacos版本到高版本，会增加管理员权限验证等等，包括会默认禁用这个接口，还有替换为Mysql等数据库，会自动禁用这个嵌入式数据库，也能防御这个漏洞，2.4.1版本之后和1.4.8版本之后还加了SQL白名单。

                  Nacos 未授权接口命令执行漏洞（CVE-2021-29442）

                  其实这个漏洞是前面这个SQL注入的进一步利用，因为前面有说，拼接的时候限制了只能进行查询操作，但是纯查询并不能RCE，所以为了进一步的利用，挖掘了一下这里的SQL查询机制，发现它可以去加载Java 方法，这就导致了可以加载外部的jar包，从而注入内存马等等。

                  • 原始影响版本：Nacos < 1.4.1
                  • 新利用（2024 公开）扩展版本：Nacos ≤ 2.4.0-BETA

                  Nacos Jraft Hessian 反序列化 漏洞（CNVD-2023-45001）

                  nacos默认的7848端口是用来处理 集群 模式下raft协议的通信，该端口的服务在处理部分jraft请求的时候使用hessian传输协议进行反序列化过滤不严，导致RCE。 影响版本:

                  • 1.4.0 <= Nacos < 1.4.6（使用cluster集群模式运行）
                  • 2.0.0 <= Nacos < 2.2.3（任意模式运行） 因为我们这里是主机测试的，所以没打算集群模式运行，当然，后续会看一下集群模式下的一些攻击。

                  public class SerializeFactory {  
      
    public static final String HESSIAN_INDEX = "Hessian".toLowerCase();  
      
    private static final Map<String, Serializer> SERIALIZER_MAP = new HashMap<String, Serializer>(4);  
      
    public static String defaultSerializer = HESSIAN_INDEX;  
      
    static {  
        Serializer serializer = new HessianSerializer();  
        SERIALIZER_MAP.put(HESSIAN_INDEX, serializer);  
        for (Serializer item : NacosServiceLoader.load(Serializer.class)) {  
            SERIALIZER_MAP.put(item.name().toLowerCase(), item);  
        }  
    }  
      
    public static Serializer getDefault() {  
        return SERIALIZER_MAP.get(defaultSerializer);  
    }  
      
    public static Serializer getSerializer(String type) {  
        return SERIALIZER_MAP.get(type.toLowerCase());  
    }  
}

                  public static RpcServer initRpcServer(JRaftServer server, PeerId peerId) {  
    GrpcRaftRpcFactory raftRpcFactory = (GrpcRaftRpcFactory) RpcFactoryHelper.rpcFactory();  
    raftRpcFactory.registerProtobufSerializer(Log.class.getName(), Log.getDefaultInstance());  
    raftRpcFactory.registerProtobufSerializer(GetRequest.class.getName(), GetRequest.getDefaultInstance());  
    raftRpcFactory.registerProtobufSerializer(WriteRequest.class.getName(), WriteRequest.getDefaultInstance());  
    raftRpcFactory.registerProtobufSerializer(ReadRequest.class.getName(), ReadRequest.getDefaultInstance());  
    raftRpcFactory.registerProtobufSerializer(Response.class.getName(), Response.getDefaultInstance());  
      
    MarshallerRegistry registry = raftRpcFactory.getMarshallerRegistry();  
    registry.registerResponseInstance(Log.class.getName(), Response.getDefaultInstance());  
    registry.registerResponseInstance(GetRequest.class.getName(), Response.getDefaultInstance());  
  
    registry.registerResponseInstance(WriteRequest.class.getName(), Response.getDefaultInstance());  
    registry.registerResponseInstance(ReadRequest.class.getName(), Response.getDefaultInstance());  
      
    final RpcServer rpcServer = raftRpcFactory.createRpcServer(peerId.getEndpoint());  
    RaftRpcServerFactory.addRaftRequestProcessors(rpcServer, RaftExecutor.getRaftCoreExecutor(),  
            RaftExecutor.getRaftCliServiceExecutor());  
      
    // Deprecated  
    rpcServer.registerProcessor(new NacosLogProcessor(server, SerializeFactory.getDefault()));  
    // Deprecated  
    rpcServer.registerProcessor(new NacosGetRequestProcessor(server, SerializeFactory.getDefault()));  
      
    rpcServer.registerProcessor(new NacosWriteRequestProcessor(server, SerializeFactory.getDefault()));  
    rpcServer.registerProcessor(new NacosReadRequestProcessor(server, SerializeFactory.getDefault()));  
      
    return rpcServer;  
}

                  这个方法简单来说就是创建并启动 Nacos JRaft 集群通信的 RPC 服务器（7848 端口），我们来看一下反序列化是在哪里执行的。 可以看到这一段，其实就是绑定上了 Hessian 反序列化工具，这样子在后面执行反序列化的时候就会直接触发。后续的反序列化代码这里就没有了，因为被封装到了GRPC 封装层里面，我们需要看依赖包才能发现。

                  Nacos Client Yaml反序列化漏洞

                  Nacos Client的Yaml反序列化漏洞主要是由于其对用户输入的处理不当所致。在Nacos中，用户可以配置各种参数和设置，这些配置通常以Yaml格式存储。当Nacos Client接收到用户的配置请求时，它会将这些Yaml数据反序列化成可执行的代码。如果攻击者能够在Yaml数据中注入恶意代码，并在适当的位置进行触发，便可能执行任意代码或造成系统崩溃。 版本要求：在1.4.2版本之下，1.4.2版本修复了。
