SSRF整理

回顾一下TOP10，整理一下

2026-03-04

一个比较迷糊的漏洞，简单说一下吧，还有一些东西，是有些忘记了，需要重新捡起来了。

利用

这个漏洞主要是一个代码逻辑的问题，所以不局限于PHP环境还是python环境还是Java环境，这里用PHP的代码来写一个示例。

<?php
// 接收用户传入的URL参数（无任何过滤）
$user_url = $_GET['url'];

// 直接使用file_get_contents请求用户传入的URL
if (!empty($user_url)) {
    // 漏洞核心：未过滤、未验证用户输入的URL，直接发起服务器端请求
    $response = file_get_contents($user_url);
    echo "请求结果：<br>";
    echo $response;
} else {
    echo "请传入url参数，例如：?url=http://example.com";
}
?>

还是很简单一个漏洞点啊。弄下来吧。

很明显可以看到，如果这个是在外网的情况下，是可以任意访问内网的资源的，当然这个说起来还是有些抽象。因为这里只显示了访问文件这一种，但是实际上非80端口的，其他端口的服务资源，都是可以使用的，因此SSRF的危害范围非常广泛。

危害

SSRF的危害还是比较简单的吧，光是能用代码访问内网资源危害及很大了。具体一点有哪些危害呢。

  1. 读取本地文件：利用file://协议读取文件
  2. 攻击内网服务：利用gopher://协议和http://协议，访问内网服务，进行攻击
  3. 绕过安全防护机制：利用安全设备对内网流量不敏感，可以进行绕过
  4. 执行恶意代码：结合其他漏洞，进一步getshell之类的

修复

对于这种漏洞的修复，在不改变原本功能的情况下，比较常见的是加上白名单，输入过滤还有协议限制。

<?php
// 定义允许访问的域名白名单
$allowed_domains = ['example.com', 'baidu.com', 'github.com'];

// 接收并过滤用户输入
$user_url = isset($_GET['url']) ? trim($_GET['url']) : '';

if (!empty($user_url)) {
    // 步骤1：解析URL，获取协议、主机名
    $url_info = parse_url($user_url);
    
    // 步骤2：验证协议（仅允许http/https）
    if (!isset($url_info['scheme']) || !in_array($url_info['scheme'], ['http', 'https'])) {
        die("错误：仅允许http/https协议！");
    }
    
    // 步骤3：验证主机名（仅允许白名单内的域名）
    if (!isset($url_info['host']) || !in_array($url_info['host'], $allowed_domains)) {
        die("错误：仅允许访问指定域名！");
    }
    
    // 步骤4：禁止访问内网IP（可选，更严格的防护）
    $ip = gethostbyname($url_info['host']);
    if (filter_var($ip, FILTER_VALIDATE_IP, FILTER_FLAG_NO_PRIV_RANGE | FILTER_FLAG_NO_RES_RANGE) === false) {
        die("错误：禁止访问内网地址！");
    }
    
    // 安全发起请求
    $response = file_get_contents($user_url);
    echo "请求结果：<br>";
    echo $response;
} else {
    echo "请传入url参数，例如：?url=http://example.com";
}
?>

这个是可以绕过的，但是还是有一定修复效果的，主要是懒了，懒得把绕过的什么都都过滤掉。
