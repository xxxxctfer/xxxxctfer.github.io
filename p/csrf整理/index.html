<!-- transfer_result.php -->
<!DOCTYPE html>
<html lang="zh-CN">
<head>
    <meta charset="UTF-8">
    <title>转账结果页</title>
</head>
<body>
    <?php
    // 1. 接收GET参数（做基础的非空验证，避免报错）
    $username = isset($_GET['username']) ? $_GET['username'] : '';
    $amount = isset($_GET['amount']) ? $_GET['amount'] : '';

    // 2. 验证参数有效性（简单校验，避免空值/非数字金额）
    if (empty($username) || empty($amount) || !is_numeric($amount)) {
        echo "<h3 style='color: red;'>参数错误！请检查用户名和金额</h3>";
    } else {
        // 3. 输出转账成功提示
        echo "<h3 style='color: green;'>向{$username}转账{$amount}元成功！</h3>";
    }
    ?>
    
    <!-- 返回转账触发页的链接 -->
    <br>
    <a href="transfer_link.html">返回转账页</a>
</body>
</html>

然后我们就可以访问看一下了。 可以看到是有一个链接可以点击的，我们点击看一下是什么情况。 可以看到直接转账出去了，虽然不是真正的转账，但是如果有一些别的和金钱联系起来的功能界面，则可以造成实际上的危害。至于修复后面再说，这里先不着急。

危害

危害其实可以看到一部分了，就是在未经操作者的同意下，直接由代码操作掉了一些危险操作。这里也只是列举一些问题所在，并没有包含全部吧。

  1. 攻击者可以通过CSRF攻击，向目标网站发送恶意请求，从而窃取用户的敏感信息
  2. 攻击者可以通过CSRF攻击，向目标网站发送恶意请求，从而发起各种恶意操作。
  3. 攻击者可以通过CSRF攻击，向目标网站发送恶意请求，破坏网站的安全。

修复

首先想明白CSRF漏洞的核心就是盗用合法用户的身份，执行危险操作的过程。所有基本上修复都是从这个点下手。

  1. CSRF Token：给每个用户的合法请求绑定一个唯一、随机、不可预测的令牌（Token），这个 Token 只存储在：服务器端（用户 Session 中）；前端页面（隐藏表单字段 / 请求头）。
  2. 验证 Referer/Origin 请求头：后端只允许来自本站域名的请求，拒绝来自其他域名（攻击者的恶意页面）的请求，从而阻止跨站伪造。
  3. SameSite Cookie 属性：Cookie 的SameSite属性可以限制 Cookie 仅在 “同站请求” 中携带，禁止 “跨站请求” 携带 Cookie。
  4. 双重验证：对敏感操作（如转账、改密码），除了常规验证，额外要求用户输入验证码 / 短信 / 密码，即使请求被伪造，攻击者也无法通过二次验证。
